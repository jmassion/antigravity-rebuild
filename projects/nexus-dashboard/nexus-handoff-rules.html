Standard Operating Procedure

Handoff Packaging Rules

How Claude.ai chat packages completed work for seamless handoff to Claude Code, Antigravity, or any execution environment.

Rule 01

📦 Standard Package Structure

Every handoff from Claude.ai chat follows the same zip structure. The receiving agent (Claude Code, Antigravity, etc.) can rely on this layout being consistent every single time.

nexus-handoff-[project]-[date]/ │ ├── 📋 HANDOFF.md ← THE KEY FILE. Human + agent readable. │ Contains everything needed to execute. │ ├── 📁 src/ ← All source files │ ├── index.html ← (or App.jsx, main.py, etc.) │ ├── styles.css ← (if separated) │ └── app.js ← (if separated) │ ├── 📁 assets/ ← Images, fonts, media (if any) │ ├── 📄 manifest.json ← Machine-readable instructions │ ├── 📄 netlify.toml ← Deploy config (if applicable) │ └── 📄 package.json ← Dependencies (if applicable)
💡 Single-file projects: If the project is one HTML file, the zip still follows this structure — the HTML goes in src/, and HANDOFF.md + manifest.json are still included. Consistency matters more than minimalism.
Rule 02

📋 manifest.json — Machine-Readable Instructions

Every handoff includes a manifest.json that any agent can parse to understand what to do without reading prose. This is the "API contract" of the handoff.

{ "handoff": { "version": "1.0", "created": "2026-03-05T18:30:00Z", "source": "claude.ai", "project": "nexus-dashboard", "description": "NEXUS Mission Control v4 with billing, compute, API/MCP views" }, "placement": { "target": "new-project", "repo": "nexus-os", "path": "apps/nexus-dashboard/", "parent_project": null, "netlify_site": "nexus-dashboard" }, "files": [ { "source": "src/index.html", "destination": "apps/nexus-dashboard/index.html", "action": "create_or_replace" } ], "dependencies": { "npm": [], "cdn": [ "https://cdnjs.cloudflare.com/ajax/libs/three.js/r128/three.min.js" ] }, "deploy": { "provider": "netlify", "build_command": "none", "publish_dir": "apps/nexus-dashboard/", "auto_deploy": true }, "post_deploy": [ "Verify all 11 tabs load correctly", "Test drag-and-drop file import", "Confirm 3D graph renders with Three.js" ], "mcp_required": ["netlify", "github"], "skills_required": [], "context_notes": "Built across 6 iterations in a single chat. v4 adds billing, compute provisioning, and API/MCP management views." }
Rule 03

🤖 Agent Prompt — Copy-Paste Ready

Every HANDOFF.md starts with a prompt block that can be directly pasted into Claude Code, Antigravity, or any agent. The agent reads it and knows exactly what to do.

# HANDOFF.md ## 🤖 AGENT PROMPT (paste this into Claude Code or your IDE agent) ``` I have a handoff package from Claude.ai chat. Here's what needs to happen: PROJECT: nexus-dashboard ACTION: Deploy new project to monorepo REPO: github.com/justin/nexus-os PATH: apps/nexus-dashboard/ Steps: 1. Create branch: feat/nexus-dashboard-v4 2. Place src/index.html → apps/nexus-dashboard/index.html 3. Place netlify.toml → apps/nexus-dashboard/netlify.toml 4. Commit with message: "feat: NEXUS dashboard v4 — billing, compute, API/MCP views" 5. Push and open PR 6. The auto-merge workflow will handle the rest The manifest.json has all file mappings and deploy config. ``` ## 📝 HUMAN SUMMARY What was built: NEXUS Mission Control dashboard v4 — a browser-based project management UI with 11 tabs including overview, kanban, 3D graph, live preview, deploys, agent queue, links, file import, billing/accounting, compute provisioning, and API/MCP management. What it needs: Just static file hosting. No build step. Single HTML file with inline CSS/JS. Three.js loaded from CDN. What changed from v3: Added billing view (credit balance, per-project costs, agent credit cards, transactions), compute view (provision Linux/Win/Mac desktops, auto-provisioning rules), and API/MCP view (key management, MCP server toggles, skills/rules editor, override matrix).
Rule 04

🏷️ Naming Convention

Predictable names so agents and humans can find things without guessing.

📛 File & Folder Naming Rules
📦
Zip filename
nexus-handoff-[project]-[YYYY-MM-DD].zip — e.g., nexus-handoff-dashboard-2026-03-05.zip
📋
Manifest
Always manifest.json at the root. Never nested.
📝
Instructions
Always HANDOFF.md at the root. Uppercase so it stands out in directory listings.
📁
Source code
Always in src/ folder. Even if it's one file.
🔀
Branch names
feat/[project]-[short-description] — e.g., feat/dashboard-billing-views
💬
Commit messages
feat: [project] — [what changed] — e.g., feat: dashboard — add billing, compute, API/MCP views
Rule 05

📍 Placement Instructions

Every handoff specifies exactly where files go in the monorepo. No ambiguity.

🎯 Placement Types
🆕
New project
Creates a new folder under apps/[project-name]/. Includes netlify.toml for auto-deploy setup.
🔄
Update existing project
Replaces specific files at their existing paths. manifest.json lists each file with "action": "create_or_replace" and exact destination paths.
📁
Add to sub-project
Targets a specific sub-folder like apps/holodeck-os/landing/. Parent project specified in manifest.
♻️
Shared component
Goes into shared/components/ or shared/utils/ for cross-project reuse.
🤖
Agent config update
Goes into agents/global/ or agents/projects/[name]/ for rules, skills, or MCP configs.
Rule 06

🔧 Dependency Declaration

The manifest declares every external dependency so the receiving agent can install what's needed before deploying.

📦
npm packages
Listed in manifest.json under dependencies.npm. If non-empty, the agent runs npm install.
🌐
CDN scripts
Listed under dependencies.cdn. Already referenced in the HTML. No installation needed — just documentation.
🐍
Python packages
Listed under dependencies.pip if applicable. Agent runs pip install.
⚙️
System tools
Listed under dependencies.system — things like ffmpeg, imagemagick, etc.
💡 Most NEXUS projects are zero-dependency. Single-file HTML with inline CSS/JS and CDN-loaded libraries. The manifest still declares the CDN dependencies for documentation, but no install step is needed.
Rule 07

🚀 Deploy Instructions

Every handoff includes explicit deploy config so the agent can ship it without asking questions.

// In manifest.json: "deploy": { "provider": "netlify", // Where it goes "build_command": "none", // or "npm run build" etc. "publish_dir": "apps/dashboard/", // What folder to publish "auto_deploy": true, // Auto-deploy on merge? "domain": "nexus-dashboard.netlify.app", // Expected URL "custom_domain": "dashboard.holodeckos.com" // If assigned } // The agent's job: // 1. Place files per manifest // 2. Commit to branch // 3. Open PR // 4. Auto-merge pipeline handles the rest // 5. Netlify webhook triggers deploy

Rule 09

✅ Verification Checklist

Every handoff ends with a checklist the receiving agent runs after deployment to confirm everything works.

🔍 Post-Deploy Verification Template
1️⃣
Site loads
Open the Netlify URL. Page renders without errors. Check browser console for JS errors.
2️⃣
All views accessible
Click every tab. Each view renders content, not blank space.
3️⃣
Interactive elements work
Buttons respond, modals open, drag-and-drop functions, chat sends messages.
4️⃣
Mobile responsive
Check on phone or resize browser. Content doesn't overflow or break.
5️⃣
No console errors
Open DevTools → Console. Zero red errors. Warnings are okay.
6️⃣
Report back
Send a summary: "Deployed [project] to [URL]. All checks pass. [Any issues found]."
Rule 10

🔄 Continuation Prompt — For New Chat Sessions

When starting a fresh Claude.ai chat to continue work, use this template. Drop in the latest file(s) and paste this prompt structure:

## Continuation prompt template: "Continuing NEXUS project. Here's the current [dashboard/component/document]. CONTEXT: - Project: [name] - Version: [v4, v5, etc.] - Last session built: [what was added] - Repo status: [deployed / not yet deployed] TASK FOR THIS SESSION: - [What you want built next] The file attached is the latest version. All code comments follow our standard: comments for a 10-year-old, emojis, ASCII art headers, TOC top and bottom. When done, package as a handoff zip with HANDOFF.md and manifest.json following our standard packaging rules."
💡 Pro tip: You don't need to re-explain the whole project. Claude has memories of your project context from past conversations. Just drop the file, state what's next, and go. The continuation prompt is mainly for precision, not for re-teaching.
